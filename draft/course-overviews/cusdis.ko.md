# Cusdis 자동화 강좌

## 강좌 개요

이 강좌는 블로그 댓글 운영을 자동화하고 싶은 초급자를 위해 설계된 실전 수업입니다. 댓글이 들어왔을 때 webhook으로 이벤트를 받고, n8n에서 워크플로우를 시작하고, Gemini로 내용을 판정한 뒤, 정상 댓글만 승인과 답글 단계로 넘기는 전체 흐름을 한 번에 연결합니다. 단순히 노드 몇 개를 배치하는 법이 아니라, 입력 계약과 분기 조건, 실패 시 대응 방식까지 함께 다루는 것이 특징입니다.

많은 입문자는 AI 자동화를 “모델을 붙이면 끝나는 것”으로 생각하지만, 실제 운영에서는 JSON 형식 강제, 오류 흡수, 승인 기준, fallback 운영 같은 세부 설계가 훨씬 중요합니다. 이 강좌는 그런 운영 감각을 초급자 수준에서 익히도록 돕습니다.

강의를 마치면 수강생은 댓글 이벤트를 자동화 흐름의 시작점으로 삼고, AI를 판정 엔진으로 사용하며, 승인/답글 API 호출까지 하나의 안전한 운영 루프로 묶는 구조를 이해하게 됩니다. 또한 공개 환경에 워크플로우를 둘 때 무엇을 확인해야 하는지도 함께 배우게 됩니다.

## 추천 수강생

- 댓글 운영을 수동으로 처리하고 있어 자동화의 첫 사례가 필요한 개발자 또는 1인 제작자
- n8n, webhook, API, JSON 개념을 실제 사례로 배우고 싶은 대학 초급 수준의 수강생
- AI를 챗봇이 아니라 운영 자동화의 판단 엔진으로 활용해 보고 싶은 사람

## 사전 준비

- HTTP 요청과 JSON 구조를 대략 이해하고 있으면 충분합니다.
- n8n Cloud 계정이나 self-hosted n8n 환경 중 하나가 있으면 실습이 수월합니다.
- Cusdis 사이트 설정 화면에 접근할 수 있으면 강의 흐름을 그대로 재현하기 좋습니다.

## 학습 목표

- Cusdis webhook을 n8n에 연결해 댓글 이벤트를 자동화의 트리거로 사용할 수 있습니다.
- Gemini가 정해진 JSON 계약으로 응답하도록 프롬프트를 설계하고 후처리 코드를 붙일 수 있습니다.
- 정상 댓글만 통과시키는 분기, 지연 처리, 승인 API 호출을 하나의 워크플로우로 묶을 수 있습니다.
- 배포 전후 점검 항목과 실패 시 복구 포인트를 운영 체크리스트로 정리할 수 있습니다.

## 상세 커리큘럼

### 1. 환경 구성과 전체 흐름 이해

첫 챕터에서는 Cusdis, n8n, Cloudflare Tunnel이 각각 어떤 역할을 맡는지 먼저 정리합니다. 자동화가 한 번에 잘 보이지 않는 이유는 도구가 많아서가 아니라, 각 도구가 어떤 책임을 지는지 섞여 보이기 때문입니다. 이 챕터는 그 경계를 분리하는 데 초점을 둡니다.

Cusdis는 댓글 이벤트를 발생시키는 원천이고, n8n은 그 이벤트를 받아 흐름을 실행하는 엔진이며, Cloudflare Tunnel은 self-hosted 환경에서 외부 신호를 받을 수 있게 해 주는 공개 경로 역할을 합니다. 도구의 역할이 보이면 이후 노드 설정이 훨씬 덜 복잡해집니다.

또한 n8n Cloud와 self-hosted를 어떻게 선택할지도 여기서 다룹니다. 초급자에게는 보통 n8n Cloud가 더 빠른 출발점이지만, self-hosted는 더 많은 통제권을 줍니다. 다만 통제권은 곧 공개 주소, 업데이트, 보안 패치, 운영 책임을 함께 가진다는 뜻이기도 합니다.

| 도구 | 주된 역할 | 없으면 막히는 지점 |
| --- | --- | --- |
| Cusdis | 댓글 이벤트를 발생시키고 운영 대상 데이터를 제공 | 자동화의 입력 자체가 생기지 않음 |
| n8n | 이벤트를 받아 조건 분기와 후속 액션을 실행 | 판정, 지연, 승인 요청이 연결되지 않음 |
| Cloudflare Tunnel | self-hosted n8n을 외부 webhook이 도달 가능한 주소로 공개 | 외부 webhook이 로컬/사설망에 도달하지 못함 |

### 2. Webhook과 트리거 만들기

두 번째 챕터에서는 Cusdis에서 댓글 이벤트를 보내고 n8n의 Webhook 노드가 그 이벤트를 받는 첫 관문을 완성합니다. 자동화는 언제나 “입력 구조를 먼저 정확히 이해하는 것”에서 시작한다는 점을 실제 사례로 보여 줍니다.

자동화에서 입력은 가장 먼저 이해해야 하는 시스템 계약입니다. 댓글 이벤트가 어떤 필드를 포함하는지, site ID는 어디에 들어 있는지, 본문과 작성자 정보는 어떤 형태인지 모르고 다음 단계를 만들면, AI 판정이나 승인 API가 뒤에서 계속 어긋나게 됩니다. 그래서 이 챕터는 단순 연결보다 payload 읽기에 더 큰 비중을 둡니다.

이 챕터를 통해 수강생은 Cusdis Site settings에서 webhook URL을 연결하고, n8n에 들어오는 payload가 어떤 형태인지 읽는 법을 배웁니다. 이후 단계에서 AI와 조건 분기를 안정적으로 붙이려면 이 입력 단계가 가장 중요하다는 메시지가 강조됩니다.

| 확인 항목 | 왜 중요한가 | 다음 단계에 미치는 영향 |
| --- | --- | --- |
| site 식별자 | 어느 사이트의 댓글인지 구분해야 함 | 승인 API 대상과 인증값 연결 |
| comment 본문과 메타데이터 | AI 판정과 답글 초안의 입력이 됨 | 판정 정확도와 후처리 안정성 |
| HTTPS 도달 가능 여부 | 외부 webhook이 실제로 도달해야 함 | 자동화 자체가 시작되느냐 여부 |

### 3. Gemini 분석과 JS 후처리

세 번째 챕터는 AI를 운영 자동화 안에서 어떻게 안전하게 쓰는지를 다룹니다. 모델이 항상 같은 형식으로 답하도록 JSON 계약을 강제하고, 응답이 깨지거나 일부 필드가 누락되더라도 워크플로우 전체가 멈추지 않도록 JavaScript 후처리를 붙이는 구조를 설명합니다.

생성형 AI를 자동화 안에 넣을 때 가장 위험한 순간은 “모델이 대충 자연어로 답했는데도 다음 단계가 계속 진행되는 상태”입니다. 사람에게는 그럴듯한 답변이더라도, 기계에게는 JSON 계약이 더 중요합니다. 이 챕터는 분류 결과와 이유, 답글 초안을 항상 같은 구조로 돌려주게 만드는 이유를 설명합니다.

프롬프트는 단순 요청문이 아니라 계약서처럼 써야 합니다. 어떤 필드가 반드시 있어야 하는지, 어떤 값만 허용되는지, 길이는 어느 정도인지, 확신이 없을 때는 어떤 상태를 반환해야 하는지를 함께 적어야 합니다. 그래야 AI가 애매한 경우에도 REVIEW 같은 중간 상태로 빠질 수 있고, 운영 리스크를 줄일 수 있습니다.

하지만 계약을 적어 두는 것만으로는 부족합니다. 실제 운영에서는 모델 출력이 깨지거나 일부 필드가 누락될 수 있기 때문에, JavaScript 후처리 코드로 파싱 실패를 흡수하고 기본값을 부여하는 방어 코드가 필요합니다.

| 필드 | 허용값 / 형태 | 왜 필요한가 |
| --- | --- | --- |
| classification | NORMAL / SPAM / REVIEW | 분기 노드가 안전하게 다음 단계 여부를 판단 |
| reason | 짧은 설명 문자열 | 운영자가 판정 근거를 추적할 수 있음 |
| replyDraft | 짧고 자연스러운 답글 초안 | 승인 단계에서 즉시 사용할 수 있는 출력이 됨 |

### 4. 조건 분기와 승인 API 호출

네 번째 챕터에서는 AI가 `NORMAL`로 판정한 댓글만 다음 단계로 보내고, 사람이 직접 답하는 것처럼 보이는 지연을 준 뒤 승인과 답글 API 호출을 수행하는 전체 흐름을 완성합니다. 자동화는 빠르기만 한 것이 아니라, 운영 리듬과 안전성을 함께 고려해야 한다는 점을 보여 줍니다.

If 노드는 이 흐름에서 가장 중요한 안전 장치입니다. Gemini가 NORMAL로 분류한 경우만 다음 단계로 보내고, SPAM이나 REVIEW는 다른 경로로 분리함으로써 자동화가 무분별하게 실행되는 것을 막습니다. Wait 노드는 즉시 반응하는 기계적인 인상을 줄이고, 운영자 입장에서도 너무 공격적인 자동화를 완화해 줍니다.

마지막으로 HTTP Request 노드는 실제 행동을 발생시키는 지점입니다. 여기서는 승인 요청과 답글 본문이 정확한 구조로 전송되어야 하고, 어느 사이트를 대상으로 하는지와 인증 정보가 올바른지도 확인되어야 합니다. 이 단계는 워크플로우가 “판단”에서 “실행”으로 넘어가는 구간이기 때문에 가장 신중해야 합니다.

| 단계 | 무엇을 하는가 | 운영상 의미 |
| --- | --- | --- |
| If | NORMAL 댓글만 다음 단계로 통과시킴 | 위험한 자동 승인을 막는 첫 안전장치 |
| Wait | 랜덤 지연으로 응답 타이밍을 늦춤 | 기계적인 반응을 줄이고 운영 리듬을 완화함 |
| HTTP Request | 승인과 답글 요청을 실제 API로 전송 | 자동화의 판단을 실제 서비스 행동으로 바꿈 |

### 5. 운영 점검과 확장 아이디어

마지막 챕터에서는 워크플로우를 실제 환경에 둘 때 어떤 노드에서 실패가 나는지, 로그를 어디서 봐야 하는지, 그리고 향후 슬랙 알림, 저장소 적재, 운영 리포트 등으로 어떻게 확장할 수 있는지를 정리합니다.

자동화는 한 번 돌아간다고 끝이 아닙니다. 공개 환경에 올리는 순간부터는 오탐, 미탐, 자격 증명 만료, webhook 장애, 모델 응답 이상, API 호출 실패 같은 현실적인 문제가 생길 수 있습니다. 따라서 마지막 챕터는 “더 멋진 자동화”보다 “문제 생겼을 때 어떻게 버틸 것인가”에 초점을 둡니다.

운영자의 관점에서 가장 중요한 것은 fallback입니다. 문제가 생겼을 때 워크플로우를 끄고 수동 승인으로 되돌릴 수 있는가, JSON 파싱 실패 로그를 별도로 모을 수 있는가, 처음 며칠간은 사람이 직접 승인 결과를 눈으로 확인하는가 같은 질문이 중요합니다.

확장 아이디어도 이 기반 위에서 나옵니다. 슬랙 알림, 댓글 아카이빙, 운영 리포트, 모더레이션 대시보드 같은 기능은 멋져 보이지만, 먼저 안전한 기본 루프가 있어야 의미가 있습니다.

| 리스크 | 처음 볼 곳 | 권장 fallback |
| --- | --- | --- |
| Gemini 출력 파싱 실패 | Code 노드 로그와 raw output | REVIEW 처리 또는 수동 승인으로 전환 |
| 승인 API 호출 실패 | HTTP Request 응답 코드와 인증값 | 워크플로우 일시 중지 후 수동 처리 |
| 오탐 / 미탐 증가 | 분류 결과 샘플과 프롬프트 계약 | 프롬프트 수정 전까지 REVIEW 비율 확대 |

## 실습 자료와 산출물

강좌의 예제 자료는 실제 워크플로우를 이해하는 순서에 맞춰 준비되어 있습니다. webhook 샘플은 입력 구조를 보여 주고, Gemini 프롬프트 문서는 모델이 따라야 하는 JSON 계약을 제시합니다. JavaScript 후처리 예제는 깨진 응답을 어떻게 흡수할지 보여 주고, 승인 요청 예시는 최종 API 호출의 형태를 확인하게 해 줍니다.

수강 후에는 전체 시스템 흐름도, 입력 payload 이해 메모, AI 판정 계약 문서, 승인 요청 body 예시, 배포 전 운영 체크리스트를 자기 서비스에 맞게 남기는 것이 목표입니다.

## 추천 학습 방법

가장 추천하는 순서는 webhook 샘플과 README를 먼저 열어 전체 흐름을 잡고, 그다음 슬라이드를 따라가며 n8n에서 노드를 다시 만들어 보는 것입니다. 처음부터 최적화하려 하지 말고, 입력 구조와 분기 흐름을 안정시키는 데 집중하는 것이 좋습니다.

실습 중에는 Gemini 계약 문서와 후처리 코드 예제를 옆에 두고, AI 응답이 항상 신뢰할 수 없다는 전제를 유지하며 따라오는 것이 중요합니다. 이 태도가 자동화를 안전하게 만듭니다.

## FAQ

### 코드를 많이 몰라도 따라갈 수 있나요?

가능합니다. 핵심은 webhook, JSON, 조건 분기라는 세 가지 개념을 실제 흐름 안에서 보는 것입니다. 자바스크립트 예제도 짧고 목적이 분명해서, 초급자도 구조를 이해하기 좋습니다.

### n8n Cloud 없이도 실습할 수 있나요?

가능합니다. self-hosted n8n과 Cloudflare Tunnel 조합으로도 전체 흐름을 재현할 수 있습니다. 강의는 두 경로를 비교하면서, 초급자가 어디서부터 시작하면 좋은지도 함께 안내합니다.

### 자동 승인이라 위험하지 않나요?

그래서 이 강의는 JSON 계약, 분기 조건, 지연 처리, 운영 체크리스트를 함께 다룹니다. 무조건 자동 승인하는 것이 아니라, 안전한 댓글만 통과시키는 운영 루프를 설계하는 데 초점을 둡니다.
